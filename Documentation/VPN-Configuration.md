# AWS Site-to-Site VPN Configuration

## 1. Overview

This document explains the implementation of the AWS Site-to-Site VPN used to establish secure communication between the on-premises office network and the AWS Virtual Private Cloud (VPC).

The VPN provides an encrypted IPSec tunnel between the Sophos XGS 136 Firewall and AWS, allowing office users to securely access the Ubuntu-based Squid Proxy Server hosted within the AWS environment.

Dynamic routing using Border Gateway Protocol (BGP) was implemented to automatically exchange network routes between AWS and the office network.

> **Note:** All IP addresses, Autonomous System Numbers (ASNs), and identifiers used in this document are examples. Production values have been sanitized.

---

# 2. VPN Objectives

The Site-to-Site VPN was implemented to achieve the following objectives:

* Establish encrypted communication between the office network and AWS.
* Securely route office traffic through the AWS-hosted Squid Proxy.
* Enable dynamic route exchange using BGP.
* Avoid exposing internal office resources to the public internet.
* Provide reliable hybrid cloud connectivity.

# 3. VPN Components

# 6. IPSec VPN

The Site-to-Site VPN uses the IPSec protocol to encrypt all traffic exchanged between AWS and the office network.

### Benefits

* Confidentiality
* Integrity
* Authentication
* Secure communication across public networks

The IPSec tunnel protects all communication between AWS resources and the office network.

---

# 7. Dynamic Routing using BGP

Border Gateway Protocol (BGP) was selected instead of static routing.

### Why BGP?

* Automatic route learning
* Automatic route advertisement
* Simplified administration
* Better scalability
* Supports future network expansion

After the VPN tunnel was established, BGP automatically exchanged routes between AWS and the Sophos Firewall.

No manual static routes were required.

# 9. VPN Validation

After deployment, the following validation checks were performed.

### VPN Validation

* VPN tunnel status displayed as Connected.
* Tunnel remained stable during testing.

### Route Validation

* BGP neighbors established successfully.
* AWS and office routes exchanged automatically.

### Connectivity Validation

* AWS resources became reachable through the VPN.
* Proxy traffic successfully traversed the encrypted tunnel.

# 11. Best Practices

The following practices were followed during implementation.

* Imported the AWS-generated VPN configuration.
* Used BGP instead of static routes.
* Restricted communication through firewall policies.
* Monitored tunnel health after deployment.
* Validated route exchange before user testing.

# 12. Outcome

The AWS Site-to-Site VPN was successfully deployed, providing secure hybrid cloud connectivity between the office network and AWS.

The encrypted VPN tunnel enabled office traffic to securely reach the Ubuntu-based Squid Proxy Server, while BGP automatically managed route exchange between the AWS VPC and the on-premises Sophos XGS Firewall.

The VPN formed the foundation for secure communication within the hybrid cloud architecture.
